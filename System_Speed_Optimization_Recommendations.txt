# Esthington CRM: System Speed & Optimization Recommendations

This document outlines the comprehensive architectural and codebase recommendations to make the CRM lightning fast across the board. They are ranked from highest impact to lowest impact.

---

### 1. Move Document Generation & Emails to Background Jobs (Biggest Culprit)
*   **The Problem:** When you create a sale or approve a payment, the system says in its comments `// Fire off background task to email Receipt...`, but the code actually uses an `await` command (`await DocumentAutomationService.dispatchDocuments(...)`). This means the user's browser is forced to physically wait while the server generates a complex PDF, uploads it to the cloud, and connects to the email provider to send it. Only after all of this finishes does the server tell the browser "Success".
*   **The Solution:** Implement a true Background Job Queue (using Redis + BullMQ or similar). The server should instantly save the sale to the database and reply to the browser in milliseconds: "Purchase recorded successfully!". Then, a separate background worker will silently generate the PDF and send the email behind the scenes without freezing the user's screen.

### 2. Parallel File Uploads
*   **The Problem:** When uploading proof-of-payment receipts, the system processes them sequentially in a loop (`for (const file of req.files) { await uploadFile(...) }`). If a user uploads 3 receipt images, the system waits for image 1 to fully upload to the cloud before starting image 2.
*   **The Solution:** Refactor this to use `Promise.all()`. This tells the server to upload all images simultaneously, which will cut upload waiting times by 50-70% for multiple files.

### 3. Implement Optimistic UI on the Frontend
*   **The Problem:** Currently, when you click a button, the frontend shows a loading spinner and completely locks up until the backend replies.
*   **The Solution:** Implement "Optimistic UI" updates. When a marketer clicks "Confirm", the frontend should immediately assume success, close the modal, show a green success toast, and update the table numbers instantly. Meanwhile, the actual network request happens quietly in the background. If it happens to fail, the UI can gracefully roll back and notify the user. This makes the app feel completely instant to the human eye.

### 4. Database Connection Pooling (PgBouncer / Prisma Accelerate)
*   **The Problem:** As your team grows, dozens of marketers and accountants will be clicking things simultaneously. Establishing a new connection to your PostgreSQL database for every request takes time. Furthermore, your backend uses heavily nested database queries (e.g., fetching a Sale, its Plot, its Estate, the Company, the Branch, and the Accountant all in one deep query).
*   **The Solution:** Ensure you are using a connection pooler like PgBouncer (or Prisma Accelerate) on your Render deployment. This keeps a pool of "warm" database connections ready to go, stripping out the connection latency for heavy queries.

### 5. Frontend Data Caching (React Query / SWR)
*   **The Problem:** Currently, every time you navigate between the "Overview" page and the "Reports" page, the frontend has to fetch all the sales data from the server all over again.
*   **The Solution:** Integrate a data caching library like React Query (or SWR). It caches the data locally in the browser. When you switch pages, it instantly loads the cached data while silently fetching any new updates in the background.
